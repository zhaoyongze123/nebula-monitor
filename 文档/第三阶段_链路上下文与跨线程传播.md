# 第三阶段：链路上下文与跨线程传播（最难点/必考点）

## 1. **TraceContext 的设计**

### TraceID 与 SpanID 的构造

在 Nebula-Monitor 中，核心是 **TraceID**（全局唯一标识链路）：

```java
// TraceHolder.java - 上下文管理的核心
public class TraceHolder {
    private static final ThreadLocal<String> traceIdHolder = new ThreadLocal<>();
    
    public static String get() {
        String traceId = traceIdHolder.get();
        if (traceId == null) {
            // 链路不存在时，自动生成全局唯一的 TraceID
            traceId = UUID.randomUUID().toString();
            traceIdHolder.set(traceId);
        }
        return traceId;
    }
    
    public static String getCurrentTraceId() {
        return traceIdHolder.get();
    }
    
    public static void set(String traceId) {
        traceIdHolder.set(traceId);
    }
    
    public static void remove() {
        traceIdHolder.remove();
    }
}
```

**设计特点：**
- **TraceID**：一旦生成就贯穿整个链路，从网关 → 业务服务A → 业务服务B → 数据库
- **自动生成**：没有 TraceID 时立即生成（懒加载式）
- **作用**：所有日志、监控都用这个 TraceID 串联，支持一条链路查询

## 2. **ThreadLocal 的局限性**

### 为什么在线程池面前会失效？

```
主线程（业务线程）                    子线程（线程池工作线程）
┌──────────────────┐                ┌──────────────────┐
│ ThreadLocal      │                │ ThreadLocal      │
│ ─────────────    │                │ ─────────────    │
│ traceId = "uuid" │ ──被线程池────→ │ traceId = null   │  ❌ 不同线程，不同 ThreadLocal 实例
└──────────────────┘                └──────────────────┘
```

**问题根源：**

```java
// 业务代码
String traceId = TraceHolder.get();           // traceId = "uuid-123"
System.out.println(traceId);                  // 输出：uuid-123 ✓

// 提交到线程池
threadPool.execute(() -> {
    String sameTraceId = TraceHolder.get();   // ❌ 返回 null！
    System.out.println(sameTraceId);          // 输出：null（重新生成新 ID）
});

// 原因：
// 每条线程都有独立的 ThreadLocal 副本
// 主线程的 ThreadLocal["traceId"] = "uuid-123"
// 子线程的 ThreadLocal["traceId"] = <未初始化>
```

**影响范围：**

| 场景 | 是否丢失 | 原因 |
|-----|--------|-----|
| 同线程方法调用 | ✓ 正常 | 同一个 ThreadLocal |
| HTTP 请求链路 | ✓ 正常 | 每个请求一个线程，HttpClientInterceptor 在主线程传递 |
| **线程池异步任务** | ❌ **丢失** | 新线程 = 新 ThreadLocal |
| **定时任务（Timer）** | ❌ **丢失** | 同理，新线程 |
| **消息队列消费** | ❌ **丢失** | 消费线程是新线程 |

## 3. **跨线程"接力"方案**

### 方案1：上下文快照 (Capture) - 【被动捕获】

```java
// 在主线程中提交前，主动抓取当前的 TraceID
String traceId = TraceHolder.getCurrentTraceId();  // 在主线程快速快照

// ⚠️问题：只做了快照，还没有传递到子线程
```

### 方案2：包装器模式 (Wrapper/Decorator) - 【手动传递】

这就是你项目中的 `TraceContextRunnable`：

```java
// TraceContextRunnable.java - 包装器核心
public class TraceContextRunnable implements Runnable {
    private final String parentTraceId;      // 【快照】在主线程中保存的 TraceID
    private final Runnable delegateTask;     // 【原始任务】真实要执行的业务逻辑
    
    public TraceContextRunnable(String traceId, Runnable task) {
        this.parentTraceId = traceId;
        this.delegateTask = task;
    }
    
    @Override
    public void run() {
        // 子线程执行时的逻辑：
        String previousId = TraceHolder.getCurrentTraceId();
        try {
            // 第1步：【恢复】把主线程的 TraceID 设到子线程的 ThreadLocal
            if (parentTraceId != null) {
                TraceHolder.set(parentTraceId);
                System.out.println("✓ 子线程继承 TraceID: " + parentTraceId);
            }
            
            // 第2步：【执行】运行真实的业务逻辑
            delegateTask.run();
            
        } finally {
            // 第3步：【清理】避免内存泄漏
            TraceHolder.remove();
            if (previousId != null) {
                TraceHolder.set(previousId);
            }
        }
    }
}
```

**工作流程：**

```
主线程：
  1. String traceId = TraceHolder.get();          // 快照：traceId = "uuid-123"
  2. Runnable wrapped = new TraceContextRunnable(traceId, myTask);
  3. threadPool.execute(wrapped);                 // 提交包装后的任务

子线程：
  1. wrapped.run() 被调用
  2. TraceHolder.set("uuid-123");                 // 【恢复】设置子线程的 ThreadLocal
  3. myTask.run();                                // 执行业务逻辑，现在能读到正确的 TraceID
  4. TraceHolder.remove();                        // 【清理】防止泄漏
```

**缺点：** 需要业务代码主动使用包装器

```java
// ❌ 易遗漏
threadPool.execute(new TraceContextRunnable(traceId, myTask));
```

### 方案3：自动增强 (Instrumentation) - 【真正的全自动】

这是你项目中用 ByteBuddy 实现的 `ThreadPoolInterceptor`：

```java
// ThreadPoolInterceptor.java - 自动拦截所有线程池调用
public class ThreadPoolInterceptor {
    
    @RuntimeType
    public static void interceptExecute(
            @This Object executor,
            @Argument(0) Runnable task,
            @SuperCall Callable<?> zuper) {
        
        // 【自动执行】无需业务代码感知
        
        // 第1步：在拦截器中自动快照
        String traceId = TraceHolder.getCurrentTraceId();
        if (traceId == null) {
            traceId = TraceHolder.get();
        }
        
        // 第2步：自动包装
        Runnable wrappedTask = new TraceContextRunnable(traceId, task);
        
        // 第3步：提交包装后的任务
        if (executor instanceof ThreadPoolExecutor) {
            ThreadPoolExecutor threadPool = (ThreadPoolExecutor) executor;
            threadPool.execute(wrappedTask);  // 提交的是包装版本
        }
    }
}
```

**工作流程：**

```
业务代码：
  threadPool.execute(myTask);                      // ✓ 完全不改代码

ByteBuddy 拦截：
  ↓ 自动调用 ThreadPoolInterceptor.interceptExecute()
  ├─ 快照 TraceID
  ├─ 包装成 TraceContextRunnable
  └─ 提交包装版本

子线程：
  TraceContextRunnable.run() → 恢复 TraceID → 执行业务
```

**对比三种方案：**

| 方案 | 方式 | 优点 | 缺点 |
|-----|-----|------|------|
| **快照** | 手动捕获 | 简单 | ❌ 只快照，没传递 |
| **包装器** | `new TraceContextRunnable()` | 清晰，可控 | ❌ 业务代码侵入 |
| **自动增强** | ByteBuddy 拦截 | ✅ **全自动零侵入** | 性能（反射）、调试复杂 |

## 4. **内存泄漏防治**

### ThreadLocal.remove() 的调用时机

**问题场景：**

```java
// ❌ 内存泄漏代码
threadPool.execute(() -> {
    String traceId = TraceHolder.get();
    // 执行业务...
    // ❌ 没有调用 remove！
});

// 后果：
// 线程池中的线程通常被复用（FixedThreadPool）
// 线程1执行完任务后被归还到线程池
// ThreadLocal<"uuid-123"> 仍然在线程1中
// 线程1再次被分配给新任务时
// 该任务会读到上一个链路的 TraceID ❌ 链路污染
```

**正确做法：** TraceContextRunnable 中的 finally 块

```java
@Override
public void run() {
    String previousId = TraceHolder.getCurrentTraceId();
    try {
        if (parentTraceId != null) {
            TraceHolder.set(parentTraceId);
        }
        delegateTask.run();
    } finally {
        // 【关键】无论异常还是正常退出，都要清理
        TraceHolder.remove();
        
        // 【高阶】如果线程池中有多层嵌套链路，需要恢复之前的 ID
        if (previousId != null) {
            TraceHolder.set(previousId);
        }
    }
}
```

**内存生命周期：**

```
线程1 执行 Task A（TraceID = "aaa"）
├─ TraceHolder.set("aaa")        【分配内存】
├─ 执行业务 A
├─ finally: TraceHolder.remove() 【释放内存】✓
└─ 线程1 归还给线程池

线程1 再被分配给 Task B（TraceID = "bbb"）
├─ 新的 TraceContextRunnable 获得 traceId = "bbb"
├─ TraceHolder.set("bbb")        【新的分配】✓ 不会读到 "aaa"
└─ ...
```

### 三层防护机制

```java
public class TraceContextRunnable implements Runnable {
    private final String parentTraceId;
    private final Runnable delegateTask;
    
    @Override
    public void run() {
        // 【防护 1】备份当前线程的旧 ID（应对嵌套场景）
        String previousId = TraceHolder.getCurrentTraceId();
        
        try {
            // 【防护 2】设置新的 TraceID（主要目的）
            if (parentTraceId != null) {
                TraceHolder.set(parentTraceId);
            }
            
            delegateTask.run();
            
        } finally {
            // 【防护 3】必须清理，避免泄漏（最关键）
            TraceHolder.remove();
            
            // 【防护 4】如果有旧 ID，恢复它
            if (previousId != null) {
                TraceHolder.set(previousId);
            }
        }
    }
}
```

**最常见的泄漏场景：**

```java
// ❌ 泄漏 1：直接忘记 remove
String traceId = TraceHolder.get();
// 使用 traceId...
// 没有调用 TraceHolder.remove()

// ❌ 泄漏 2：异常导致 remove 未执行
String traceId = TraceHolder.get();
try {
    throw new RuntimeException("出错");  // 异常
} catch (Exception e) {
    // ❌ 没有在 finally 中 remove
}

// ❌ 泄漏 3：多层包装的 ID 污染
// 线程执行被嵌套的 Task，未清理外层
threadPool.execute(new TraceContextRunnable("id-outer", () -> {
    threadPool.execute(new TraceContextRunnable("id-inner", () -> {
        // 使用 id-inner
    })); // ❌ 如果内层没有 remove，外层 remove 时会清掉 id-outer
}));

// ✓ 正确做法：始终使用 try-finally
```

## 📊 完整时序图

```
【时刻 1】主线程 - HttpRequest 进来
  ├─ HttpClientInterceptor 拦截
  └─ TraceHolder.get() → 生成 "trace-001"

【时刻 2】主线程 - 业务代码调用线程池
  ├─ 业务代码：threadPool.execute(myTask)
  ├─ ByteBuddy 拦截：ThreadPoolInterceptor.interceptExecute()
  │  ├─ snapshot: traceId = TraceHolder.get() → "trace-001"
  │  └─ wrappedTask = new TraceContextRunnable("trace-001", myTask)
  └─ 提交 wrappedTask 到线程池

【时刻 3】子线程 - 执行链路

  子线程启动 wrappedTask.run()
  ├─ previousId = TraceHolder.get() → null
  ├─ TraceHolder.set("trace-001")  ✓ 恢复链路 ID
  ├─ delegateTask.run()            ← 业务代码在这里执行
  │  └─ 日志系统读 TraceHolder.get() → "trace-001" ✓
  └─ finally:
     ├─ TraceHolder.remove()       ✓ 清理
     └─ 无需恢复（previousId 为 null）

【时刻 4】子线程完成
  └─ 线程归还线程池，准备执行下一个任务
```

## 🎯 核心考点总结

| 考点 | 答案 | 关键词 |
|-----|------|--------|
| 为什么需要 TraceID？ | 跨服务链路追踪，完整可观测性 | Global ID, 全链路 |
| ThreadLocal 为什么失效？ | 新线程 = 新 ThreadLocal 副本 | 线程隔离 |
| 怎么解决线程池问题？ | 快照 + 包装器 + 恢复 | Capture → Wrap → Restore |
| ByteBuddy 的作用? | 自动拦截线程池调用，零侵入 | Instrumentation |
| remove() 什么时候调用? | finally 块，无条件执行 | 防泄漏，必须 |
| 嵌套链路怎么处理? | backup & restore previousId | 多层保护 |
